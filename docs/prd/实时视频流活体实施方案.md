# 实时视频流活体实施方案

## 1. 目标

- 在当前“静态样本活体检测”基础上，进入下一阶段的门禁机实时视频流活体能力建设
- 面向老款安卓门禁机场景，优先保证：
  - CPU 可运行
  - 响应稳定
  - 接口简单
  - 误触发可控

## 2. 当前基础

- Python 服务已具备：
  - 人脸检测
  - 静默活体单图检测
  - 活体批量样本测试
  - 识别前置活体拦截
- 当前未具备：
  - 连续视频帧输入
  - 多帧活体融合判断
  - Android 端实时帧采集与抽帧上传
  - 动作活体引导

## 3. 推荐分阶段实施

### 3.1 第一阶段：伪视频流验证

- 目标：在 Python 服务内模拟“连续帧判定”
- 输入：
  - 连续图片序列
  - 或从测试视频中抽帧后的图片集
- 输出：
  - 单帧活体结果
  - 多帧融合结果
  - 通过率和抖动情况
- 作用：
  - 在不接 Android 的情况下先验证多帧逻辑

### 3.2 第二阶段：Java 中转接口

- 目标：定义 Android -> Java -> Python 的实时帧调用口径
- Android 负责：
  - 摄像头预览
  - 固定频率抽帧
  - 上传关键帧
- Java 负责：
  - 设备鉴权
  - 请求编排
  - 打卡业务主控
- Python 负责：
  - 多帧活体判断
  - 活体通过后触发识别

### 3.3 第三阶段：门禁机实机联调

- 目标：在老款安卓门禁机上验证性能与交互
- 验证重点：
  - 帧率
  - 上传频率
  - 网络延迟
  - 活体稳定性
  - 弱光场景表现

## 4. 实时视频流推荐链路

```text
Android 摄像头预览
  -> 固定频率抽帧
  -> Java 接口鉴权与转发
  -> Python 多帧活体判断
  -> 活体通过
  -> 人脸识别
  -> Java 联合业务判定
  -> 门禁机展示结果
```

## 5. Android 端建议

### 5.1 采集方式

- 推荐 `Camera2`
- 老设备可兼容旧版 Camera API
- 预览分辨率优先选择中低档位，避免过重

### 5.2 抽帧策略

- 不建议整段视频持续上传
- 建议：
  - 预览流本地持续运行
  - 每 `200ms - 400ms` 抽一帧
  - 每次活体会话上传 `3-5` 帧

### 5.3 会话控制

- 每次用户站定后创建一次 `traceId`
- 一次活体会话内复用该 `traceId`
- 结果返回后结束会话

## 6. Python 端建议

### 6.1 新增能力

- 新增多帧活体接口，例如：
  - `POST /api/face/liveness-sequence-check`
- 输入：
  - `traceId`
  - 多张帧图片
  - 可选设备信息
- 输出：
  - 单帧结果列表
  - 最终融合结果
  - 建议是否进入识别

### 6.2 多帧融合策略

- 推荐先用轻量规则：
  - 至少 `N` 帧中有 `M` 帧活体通过才判定通过
- 示例：
  - 5 帧中至少 3 帧 `PASS`
- 优点：
  - 实现简单
  - 便于观察
  - 容易调试阈值

### 6.3 识别触发策略

- 活体通过后再提取最优帧做人脸识别
- 不建议每一帧都做完整识别
- 推荐从通过帧中挑选：
  - 人脸最大
  - 清晰度更高
  - 姿态更正

## 7. Java 端建议

- Java 继续做主控层
- 建议职责：
  - 设备鉴权
  - 参数管理
  - 会话管理
  - 识别与打卡联合判定
  - 审计日志
- 不建议把活体模型直接放到 Java 中执行

## 8. 接口建议

### 8.1 Android -> Java

- `POST /api/device/face/session/start`
- `POST /api/device/face/frame/upload`
- `POST /api/device/face/session/finish`

### 8.2 Java -> Python

- `POST /api/face/liveness-sequence-check`
- `POST /api/face/recognize`

### 8.3 返回字段建议

- `traceId`
- `livenessResult`
- `livenessScore`
- `passedFrameCount`
- `totalFrameCount`
- `bestFrameIndex`
- `recognizeResult`
- `similarity`
- `message`

## 9. 性能建议

- Python 服务先走 CPU
- 单次活体会话建议控制在 `3-5` 帧
- 单帧大小建议压缩后上传
- 端到端目标：
  - 活体 + 识别控制在 `1.5s - 2.5s`

## 10. 测试建议

### 10.1 实验室阶段

- 真人连续帧
- 打印照片连续帧
- 手机翻拍连续帧
- 弱光连续帧
- 侧脸连续帧

### 10.2 实机阶段

- 老门禁机 CPU 占用
- 内存占用
- 网络抖动
- 连续多人排队
- 重复触发抑制

## 11. 风险点

- 老设备摄像头预览帧质量不稳定
- 网络上传频率过高会导致延迟上升
- 多帧活体会增加整体时延
- 实际门口光照变化会影响稳定性

## 12. 建议下一步

- 先在 Python 侧补一个“多帧活体序列接口”的 Demo
- 再定义 Java 中转接口
- 最后再接 Android 摄像头实时抽帧联调
