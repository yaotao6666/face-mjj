# 浏览器替代方案-本地实时活体联调计划

## Summary

- 目标：在不使用浏览器打开摄像头的前提下，为当前 `python-face-demo` 提供一个可落地的实时联调方案，用于局域网环境下验证“摄像头采集 -> 活体检测 -> 人脸识别”链路。
- 结论：本轮优先选择 `PC 本地摄像头采集工具`，而不是 H5、RTSP 直拉或 Android/Java 先行接入。
- 方案定位：
  - 作为实时联调与算法验证工具
  - 直接复用现有 Python HTTP 接口
  - 不替代最终门禁机 APK
  - 不在本轮引入多帧融合新接口
- 成功标准：
  - 本地 USB 摄像头可正常打开并预览
  - 工具可按固定频率抽帧并调用现有 `/api/face/recognize`
  - 终端或预览窗口可展示活体结果、识别结果、相似度、耗时
  - 支持配置摄像头编号、服务地址、抽帧间隔

## Assumptions & Decisions

- 决策：本轮不继续以 H5 为主线。
  - 原因：浏览器方案依赖安全上下文、权限策略和宿主浏览器兼容性，局域网联调成本更高。
- 决策：本轮优先采用 `PC 本地摄像头采集 + Python 接口调用`。
  - 原因：实现速度最快，依赖最少，最适合先验证“实时采集替代浏览器”的可行性。
- 决策：首版直接调用现有 `POST /api/face/recognize`。
  - 原因：该接口已整合活体与识别，能最快形成完整闭环。
- 决策：本轮不新增后端接口，不改动 `app/api/face.py` 和数据结构。
  - 原因：用户当前问题是“是否还有其他方式”，当前重点是补测试入口，不是扩展后端能力。
- 决策：本轮不实现真正的多帧融合活体，仅做“单帧周期抽样”。
  - 原因：与当前后端能力匹配，风险最低，便于先验证实时采集链路。
- 决策：将服务地址、摄像头编号、抽帧间隔做成脚本参数，而不是写死。
  - 原因：当前服务端口历史上有过 `8000/8001/8002` 的切换，参数化更适合联调。
- 假设：测试机存在可用本地摄像头，且允许被 OpenCV 打开。
- 假设：局域网内的 Python 服务地址由操作者启动时指定，例如 `http://127.0.0.1:8002`。

## Alternatives Considered

- 方案一：Android 原生测试壳
  - 优点：最接近最终门禁机场景
  - 缺点：当前开发成本更高，设备兼容和打包调试链路更长
  - 结论：留到后续门禁机联调阶段
- 方案二：RTSP/IP 摄像头 + 转码桥接
  - 优点：适合固定机位、局域网摄像头场景
  - 缺点：需要额外流媒体桥接层，超出当前最小验证范围
  - 结论：不作为当前第一优先级
- 方案三：PC 本地摄像头采集工具
  - 优点：实现最轻、闭环最快、便于快速调阈值
  - 缺点：不等于最终交付形态
  - 结论：作为当前实施方案

## Proposed Changes

### 1. 新增接口调用封装

- 文件：`e:\other\local-rl\python-face-demo\scripts\live_client.py`
- 目标：把图像编码与 HTTP 调用从摄像头逻辑里拆分出来，形成可复用的联调客户端。
- 实现方式：
  - 提供一个轻量客户端类或函数，接收 `numpy` 的 BGR 帧
  - 使用 OpenCV 将帧编码为 JPEG 二进制
  - 以 `multipart/form-data` 调用 `POST /api/face/recognize`
  - 可选支持 `threshold` 参数透传
  - 统一返回解析后的 JSON 数据与错误信息
- 关键行为：
  - 接口超时要可配置
  - 服务不可达时返回清晰错误
  - 响应不是 200 时保留后端错误原文

### 2. 新增本地实时采集脚本

- 文件：`e:\other\local-rl\python-face-demo\scripts\live_camera_demo.py`
- 目标：从 PC 摄像头实时采集画面，按固定频率抽帧，请求现有后端识别接口，并把结果实时展示出来。
- 实现方式：
  - 使用 OpenCV 打开摄像头设备
  - 启动本地预览窗口
  - 主循环中持续读取预览帧
  - 按 `interval-ms` 控制请求频率，避免每帧都发请求
  - 调用 `live_client.py` 将当前帧发送给后端
  - 在画面上叠加最新一次结果
- 参数设计：
  - `--server`：后端地址，如 `http://127.0.0.1:8002`
  - `--camera`：摄像头编号，默认 `0`
  - `--interval-ms`：抽帧间隔，默认建议 `500`
  - `--threshold`：可选识别阈值覆盖
  - `--width` / `--height`：可选采集分辨率
- 展示内容：
  - 连接状态
  - `livenessResult`
  - `livenessScore`
  - `matched`
  - `employeeNo`
  - `employeeName`
  - `similarity`
  - `elapsedMs`
  - 最近错误信息
- 交互要求：
  - 支持按 `q` 退出
  - 摄像头打开失败时立即退出并提示原因
  - 服务调用失败时不中断预览，只提示错误并继续下一轮抽帧

## Verification Steps

1. 启动 `python-face-demo` 服务，先确认 `GET /health` 正常。
2. 调用一次 `POST /api/gallery/precheck` 与 `POST /api/gallery/rebuild`，确认底库可用。
3. 使用现有图片先手工调用一次 `POST /api/face/recognize`，确认后端识别链路正常。
4. 运行 `scripts/live_camera_demo.py`，确认本地摄像头可被正常打开。
5. 观察预览窗口，确认在设定的抽帧间隔下持续收到识别结果。
6. 真人站到镜头前，确认可看到 `livenessResult=PASS` 且命中员工信息。
7. 使用照片翻拍或屏幕翻拍测试，确认能观察到活体结果差异。
8. 修改 `--server`、`--camera`、`--interval-ms` 参数，确认参数均生效。
9. 关闭后端服务后再次运行或联调，确认脚本能持续提示错误但不异常崩溃。
10. 退出脚本后确认摄像头资源被释放，可再次正常打开。
