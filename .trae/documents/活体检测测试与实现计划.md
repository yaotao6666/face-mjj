# 活体检测测试与实现计划

## Summary

- 目标：在现有 `python-face-demo` 基础上，先落地一版 `Python 服务侧静默活体检测` 能力，并形成一套可执行的测试方案，用于验证“真人通过、照片/屏幕翻拍尽量拦截”的基础效果。
- 范围：
  - 先做 `Python 服务` 侧活体能力，不接入 Android 门禁机交互，不接 Java 服务联调。
  - 先做 `静默活体优先`，暂不实现动作活体链路。
  - 测试先覆盖 `真人样本 / 纸质照片翻拍 / 屏幕翻拍` 三类基础样本。
  - 下一步先做 `样本测试`，不直接接入实时摄像头视频流。
- 成功标准：
  - 新增独立活体检测接口或在识别接口中输出活体结果。
  - 能对基础样本集输出 `PASS / REJECT` 及分值。
  - 能形成最小测试数据组织方式、执行流程和结果记录方式。
  - 为后续接入 Android 和 Java 保留统一返回结构。

## Proposed Changes

### 1. 引入活体检测能力的推荐路线

- 推荐方案：在 `python-face-demo` 中接入 `预训练静默活体模型`，由服务端完成：
  - 人脸检测
  - 人脸裁剪
  - 活体分值推理
  - 活体通过/拒绝判定
- 推荐模型形态：
  - 优先选 `ONNX` 形式的轻量反欺骗模型
  - 要求可在 `Windows + CPU` 环境运行
  - 模型文件落在 `python-face-demo/models/` 下，与当前人脸检测/识别模型保持一致
- 推荐原因：
  - 与当前 Demo 技术栈最兼容
  - 部署轻量，不依赖 GPU
  - 便于后续挂接到 `recognize` 链路

### 5. 活体测试方案

#### 5.1 样本组织

- 真人样本：
  - `data/liveness/real/`
  - 要求同一人至少 3-5 张不同角度或光照图片
- 纸质照片翻拍样本：
  - `data/liveness/photo_spoof/`
  - 将人脸照片打印后再用手机或摄像头翻拍
- 屏幕翻拍样本：
  - `data/liveness/screen_spoof/`
  - 将人脸照片在手机/平板/显示器上展示后再拍摄

#### 5.1.1 为什么先不直接做实时视频流

- 当前阶段目标是先验证“活体模型是否可用”和“阈值是否合理”
- 如果一开始就做视频流，会同时引入：
  - 摄像头采集
  - 连续抽帧
  - 多帧时序判定
  - 延迟和性能问题
- 这些问题会和活体模型本身的效果混在一起，不利于快速定位问题
- 因此当前计划先做静态样本或抽帧样本测试，后续再单独规划实时视频流活体

#### 5.2 测试目标

- 真人样本尽量判为 `PASS`
- 纸质照片翻拍尽量判为 `REJECT`
- 屏幕翻拍尽量判为 `REJECT`

#### 5.2.1 后续实时视频流阶段

- 在样本测试跑通后，再进入第二阶段：
  - 接摄像头视频流
  - 固定频率抽帧
  - 连续多帧做活体判断
  - 观察稳定性、误判率和实时延迟
- 这一阶段不纳入本轮实现计划

#### 5.3 基础验收口径

- 先不追求严格生产级指标
- 先验证：
  - 活体模型可加载
  - 单图活体结果可返回
  - 三类样本能跑通并得到统计结果
  - 可以通过阈值调节观察 `PASS/REJECT` 变化

#### 5.4 测试结果输出建议

- 输出字段：
  - `samplePath`
  - `sampleType`
  - `livenessScore`
  - `livenessResult`
  - `expectedResult`
  - `isCorrect`
- 批量测试建议额外输出：
  - `accuracy`
  - `real_pass_rate`
  - `photo_reject_rate`
  - `screen_reject_rate`

### 6. 服务实现建议

- `face_engine.py`
  - 继续负责人脸检测和对齐
  - 为 `liveness_service` 提供统一的人脸裁剪输入
- `liveness_service.py`
  - 负责加载反欺骗模型
  - 对裁剪后的人脸进行活体推理
  - 返回分值与判定结果
- `liveness_test_service.py`
  - 负责遍历测试样本目录
  - 记录逐图结果与总体统计
- `face.py`
  - 新增活体检测接口和批量测试接口
  - 后续可将活体结果并入 `recognize`

## Assumptions & Decisions

- 决策：下一步先在 `Python 服务` 实现活体检测
  - 原因：现有 Demo 已跑通识别链路，在此处扩展活体成本最低
- 决策：首期先做 `静默活体`
  - 原因：与当前“服务端先验证算法能力”的目标一致，不需要立即引入 Android 交互动作
- 决策：测试先覆盖 `真人 / 纸质照片翻拍 / 屏幕翻拍`
  - 原因：这是基础活体最核心的三类样本
- 决策：先做样本测试，不直接做实时视频流
  - 原因：先验证模型和阈值，再处理视频流、时序和性能问题，风险最低
- 决策：活体能力先以独立接口存在
  - 原因：便于单独调试、单独测试、单独调阈值
- 决策：活体阈值同样落在 `YAML` 配置中并支持热生效
  - 原因：与当前检测阈值、识别阈值管理方式保持一致
- 假设：允许引入额外 ONNX 模型和相应运行时依赖
- 假设：当前阶段接受“基础可用”优先于“最终高精度”

## Verification Steps

1. 在 `models/` 下放置活体 ONNX 模型，并通过服务成功加载。
2. 调用 `POST /api/face/liveness-check` 上传真人图片，确认返回 `livenessScore` 与 `livenessResult`。
3. 对照片翻拍图片调用同一接口，确认能返回拒绝结果或明显较低分值。
4. 调用 `POST /api/face/liveness-batch-test`，扫描 `data/liveness/` 样本集并输出统计结果。
5. 调整 `settings.yaml` 中的 `liveness_threshold`，确认接口无需重启即可使用新阈值。
6. 选择一批真人/攻击样本，对结果做人工抽查，确认基础判断方向正确。
7. 在确认单独活体效果后，再评估是否并入 `recognize` 链路。
